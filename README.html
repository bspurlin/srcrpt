SrcRpt (or "source report") is a tool for the scanning and analysis of a body of source code based on keywords and keyphrases.

Srcrpt and its derviatives have been successful because SrcRpt introduced the concept of the Uniqline.


SrcRpt is used in two separate phases: the scan and the analysis. The scan can take place on any platform on which Perl can be installed. The analysis takes place using Internet Explorer 6.0 or above on MS Windows, using the XML/XSLT and HTML Application features of IE. Binaries are not scanned, although a list of binaries that SrcRpt encounters is placed in the Headers report.

Usually the scan and the analysis are carried out by different groups. No software has to be installed for the analysis. A utility is provided to assist in zipping up the scan and its auxiliary files for transmission from the group doing the scan to the group doing the analysis.

HOW TO SCAN SOFTWARE (go here for SrcRpt scanner instructions and command line options)      Top

Attorneys often require that software be scanned - the lawyers will not be "comfortable" with a piece of software until it has been scanned and the scan has been analyzed.  Although the lawyers themselves will readily admit that they are never really completely "comfortable" with any software, their level of "comfort" will be increased if the analysis of the scan shows that the software is of low risk.

There are certain areas that are associated with high risk:
  • Redistribution of General Public License (GPL) and Lesser General Public License (LGPL) licensed software, and other software that requires redistribution under the original license.
  • Meeting or failing to meet third party licensing requirements when redistributing software.
  • Redistributing software that is of unknown or suspect pedigree;  e. g., Open Source software for which the authors did not have sufficient rights to make the contribution to Open Source.
  • Unknowingly redistributing software that is of third party origin.

In all situations involving source code scanning it is best to keep them overall goal in mind:  reducing risk.  With risk reduction in mind, the most important of the scanning topics below is "What to scan".

SrcRpt was designed to assist attorneys in their efforts to reduce the risks of redistributing Open Source Software, and in the process of performing Due Diligence when contributing software to Open Source, acquiring software companies and divesting software.  Some companies require all proprietary code to be scanned before a product may be shipped. All of these business activities involve some risk.  Source code scanning is one tool to analyze risk. There is a variety of commercial software available to accomplish some of the same ends as SrcRpt, e.g. Black Duck ProtexIP. 

Keyword or keyphrase scanning will not produce meaningful results from scanning binaries and will not produce hits on keywords or phrases not scanned for.  The sentence "The following code was stolen from Richard Stallman" will not produce a hit from the sample keyword list that accompanies this distribution. Anyone concerned about such constructs had better add the keyword "from", "stolen" or "stole!" to the keyword list. Another limitation of keyword scanning is the large number of spurious hits ("clutter" or "noise") sometimes encountered.  The keyword "from" will significantly increase the clutter, but is useful when  scanning smaller bodies of code.  See The Keyword/Keyphrase File

What to Scan           

Top

These remarks apply to distribution or redistribution of software, not acquisitions.

Keep in mind that the purpose of scanning is to reduce risk.  When redistributing third party binaries or distributing binaries buiilt from original code the risk is in the binaries; what goes out the door.  Items that do not ship, or do not build into binaries that ship, need not be scanned.

There are three techniques used with SrcRpt to limit scanning to the body of source code from which binaries are built


1.  Scanning from a list of files using the -I <file of file paths> option.

2.    Eliminating files by their suffixes/prefixes using the -X <prefix,...> and -x <suffix,...> options.

Following are some examples of  SrcRpt command lines using the -x and -X options:

In this example build-only files (Makefiles; build related scripts) and test only files (with the extension phpt) are not scanned.

3.   Eliminating directories - paring down a directory tree by deleting, for scanning purposes, subdirectories that are not built - optional features, contributions, extensions, etc.This may be done on the command line with the -y switch with a comma-delimited list of directories as an argument, e. g., "-y xdocs,STATUS"

Cooperation with Release Engineering is essential to knowing that the pared-down directory tree will actually build objects that are shipped.

Ad Hoc Keyword Lists            Top



It is often advantageous to create a keyword list for a special purpose.  Suppose, for example, a diff-scan shows a bunch of author names. Were those authors present in the basis code?  Create a keyword list of the author surnames and run it against the original base of code to determine if the authors are unique to the later code, or not.

In patent-related investigations it can be useful to create a keyword list related to the technology area under investigation,  "lossy", "lossless", "codec" and "huffman" might be useful keywords in a patent investigation of compression software.

Regular Expressions     

            Top
See The Keyword/Keyphrase File. An example of a regular expression:
copyright(?!.{1,48}yoyodyne|.{1,48}international) QQ
The above expression is intended to detect any non-Yoyodyne International copyright statement.

Another example:
\(c\)|.\#169\;|.copy\;|© QQ
This expression is meant to find all copyright statement related symbols in addition to the familiar "(C)".  Note that certain characters have to be escaped with "\".

Elimination of keywords from the standard list     

Top

When scanning very large bodies of code, or under time pressure, it may be advantageous to create a shorter keyword/keyphrase list using the standard list as a basis.  Do this in consultation with the IP attorney on the project.  See The Keyword/Keyphrase File for suggestions on which keywords are of lower priority. 

The Copyright Holder scan

Top

This report can greatly speed the identification of  copyright holders for outbound licensing requirements or in acquisitions, particularly for larger bodies of code.

The Copyright Holder scan attempts to define a grammar of copyright statements, and recursively parse all code contexts containing the keyword "copyright!" or any of three copyright symbols based on that grammar.

The Copyright Holder scan is generated separately from a normal SrcRpt scan, with the -r switch in place of the -f <keyword file> switch.  No keyword file is used for the Copyright Holder report.  See here for an example of a Copyright scan command line.  Following is an example of the top level copyright holder report for a large subset of PHP 5.1.2:

Do not try to combine Copyright Holder scans with keyword scans. They must be separate scans.

Q: Does the keyword scan pick up everything that the Copyright Holder scan does?

A: The keyword scan should pick up everything that the copyright holder scan does. The copyright holder scan is meant for larger bodies of code when the results of the keyword scan are so voluminous that it becomes difficult to analyze without missing something.

The copyright holder scan is meant as a top-level view of a large body of code. It is meant to supplement the keyword scan. It is meant for use particularly in acquisitions when there is a time constraint.

The copyright holder scan may allow you to find something in a large body of code that you might miss with the keyword scan.

For small bodies of code, use only the keyword scan.

Scanning proprietary code

         Top
Probably the company's copyright statements are not of interest.  Use the regular expression in the above example to find non-Yoyodyne copyright statements.  Apply similar regular expressions to the "all rights reserved" and "(c)" keywords.

A keyword list used to clear Open Source is not necessarily the best for scanning proprietary code.  It will assist in finding  Open Source that has crept into nominally proprietary code, but there may be other third party code or other legal issues that need to be scanned for.  Consult with Intellectual Property law before scanning proprietary code to see if there is a need to scan for anything other than Open Source.

Some groups may require ongoing scanning of proprietary code.  The SrcRpt diff scan option may be helpful in such cases.  Consult with the author about enhancements to the diff scan capability that in the future will allow diffs to be scanned without the original source tree being present.

The Uniqline


The central metaphor of SrcRpt is the Uniqline - probably better called "unique context": a block of source code that can occur more than once in a group of files being scanned. SrcRpt assumes that a Uniqline has to be marked as "Done" or "Active" only once; that it has to be analyzed only once, and that the analysis on a Uniqline, once done, apples to the entire body of code being analyzed. From this assumption derives the advantage of SrcRpt over a file-by-file analysis. Once a Uniqline is analyzed, it is analyzed everywhere.

In the Keywords report (The "ULs" and "Tot" columns display the number of distinct or Unique lines associated with each keyword, and the total number of such Uniqlines in the code scanned).  In the SrcRpt analyzer (but not in the example table above) clicking on a keyword in the left hand column will open a report on unique lines containing that keyword.

Gain over grep (GOG) represents the gain achieved by using SrcRpt compared to a file-by-file listing of the hits in each file. In the above example the average Uniqline appears 10.61 times in the source code being scanned. Hopefully each uniqline has to be checked off or marked as done only once. In a file by file approach the uniqline would have to be checked off in each file, more than ten times.

Compare two scans

Top

Quick Start: Try File Menu : File->Compare. Note the "Compare Only Analyzed" checkbox

Script Method

Performing a diff scan (see below) with SrcRpt or CSAT has required both source code trees to be present. The Compare facility of SrcRpt compares the XML produced by two scans, and does not require source code to be present.

Compare may be used from the File->Compare menu or from a separate Perl utility.

Example: Use the command line utility to compare scans of openssl-0.9.7j and openssl-fips-1.1.2 , which is stated to be a FIPS 140-2 validated version or subset of OpenSSL.

First do two SrcRpt scans of the two source trees (or use a previously available scan of one of them):

perl source_report.pl -d openssl-0.9.7j/ -f keywords.txt -o openssl-0.9.7j.scan.1.xml
perl source_report.pl -d openssl-fips-1.1.2 -x Makefile -f keywords.txt -o openssl-fips-1.1.2.scan.1.xml

Now pass the two xml filenames to a short script calling the new Compare module:

perl -e 'use Compare;$x=Compare->new("openssl-0.9.7j.scan.1.xml","openssl-fips-1.1.2.scan.1.xml");$x->parse1();$x->parse2()'

The output is a new xml file called "openssl-0.9.7j.scan.1.compare.openssl-fips-1.1.2.scan.1.xml". The new xml file refers to the source code in the second distribution, openssl-fips-1.1.2 and has ULs marked as "not Interesting" if they appear in the first distribution, openssl-0.9.7j .

When opened with SrcRpt, the analyze report of "openssl-0.9.7j.scan.1.compare.openssl-fips-1.1.2.scan.1.xml" looks like this, showing that all hits in fips-1.1.2 are also found in openssl-0.9.7j:

Diff Scans

Top

The following color scheme is used in the unique-line reports and the file reports:

Distribution 1
Distribution 2
Both Distributions




In the unique line reports Distribution 1 color or Distribution 2 color means that the unique line is present and has changed (been removed or added) only in that distribution. The Both Distributions color indicates that a unique line has changed in at least one file in both distributions. Usually the Both Distributions color means that a file has been moved from one directory to another between distributions.

Distribution 1 color hits may be turned off or rendered invisible from the View menu. They are still there, however, and their state will influence the Analysis report.

In the file oriented reports the colors indicate that a file is found only in one or the other distribution. No color indicates that a file of a given name and path is present in both distributions

The directory structure of the two distributions must be identical, or nearly so, for the diff scan to be useful. Ideally the directory structure should differ only in the top-level directory names, e. g., "lucene-1.3-final" vs. "lucene-1.4-final".

Perl 5.6 or greater must be installed. Perl 5.8+ is highly recommended. SrcRpt will scan under Windows, Linux or Unix. Iseries and z/OS code has also been scanned. Unzip the SrcRpt distribution into a folder immediately above the folder containing the source to be scanned.

Usage:                                                        Top


perl source_report.pl -h
perl source_report.pl -r -d <search dir> [ -o <output file> ]
perl source_report.pl -f <keyword file> -d <search dir> [ -o <output file> ]
perl source_report.pl -f <keyword file> -d <search dir> -c <diff dir> [ -o <output file> ]

In addition the following options may be used:

[ -n ] [ -x <exclude suffix>][,... ] [ -X <exclude prefix>][,... ] [ -y <exclude directory>][,... ] [ -t <number of trailing characters> ] [ -q ]

Where -n turns off the pseudo-keywords "URL" and "email", -x and -X are used to exclude filenames matching the specified suffixes and prefixes, -y excludes any directory of the specified name (and all subdirectories) and -t specifies the number of trailing characters to include by default after a keyphrase expression match.

With "-r" (the Copyright Holder report), "-n" is set by default, turning off the pseudo-keywords "URL" and "email".

The -q option allows the creation of a scan directory, with or without chunked files, for compression and transmission.

In the following example, a keywords file "keywords_2004_03_29.txt" will be used to scan all non-binary files in a source tree beginning at directory lucene-1.3-final with XML output. The output file will be named (in the absence of the -o switch) "2.out.xml".

perl source_report.pl -f keywords_2004_03_29.txt -d lucene-1.3-final

In this example a report on the diffs between two Perl versions is produced:

perl source_report.pl -f keywords.txt -d perl-5.8.3 -c perl-5.8.6 -o 2.out.perl5.8.3.perl5.8.6.diff.2.xml

In the following example using the keywords file "keywords_2005_07_28.txt", a diff scan between two versions of Jakarta log4j is produced ignoring xml and html files:

perl source_report.pl -f keywords_2005_07_28.txt -d jakarta-log4j-1.1.1 -c jakarta-log4j-1.1.3 -x xml,html -o log4j-1.1.1.diff.1.1.3.srcrpt.2.xml

This example forces 80 characters of context after every keyphrase hit and excludes all files beginning with "Change" as well as all directores named "doc".

perl source_report.pl -d TimeDate-1.16 -f keywords_2005_07_28.txt -t 80 -o TimeDate-1.16.srcrpt.5.xml -X Change -y doc

Here the copyright scan is run on a directory called "build.buildforge.1", excluding filenames with certain prefixes and suffixes:

perl source_report.pl -r -d build.buildforge.1 -X Makefile -x phpt,.mk,makefile,.sh,ignore,.in -o php.build.buildforge.copyr.4.srcrpt.xml

 
To scan source: Top

Open a command window and change directory to the directory where SrcRpt has been unzipped.

If the source code is found in the folder "lucene-1.3-final" and a keyword/keyphrase file "keywords.txt" exists the command line would be:

perl source_report.pl -d lucene-1.3-final -f keywords.txt

The order on the command line makes no difference and the switches are not case sensitive.

perl source_report.pl -F keywords.txt -d lucene-1.3-final

is exactly equivalent to the above.

perl source_report.pl -f keywords.txt -d lucene-1.3-final -n

is equivalent to the above except reports on email addresses and URL's (pseudo-keywords) are not prepared.

Sometimes it is useful to specify the name of the output file, rather that accepting the default ("2.out.xml"). A filename must end in *.xml:

perl source_report.pl -f keywords.txt -d lucene-1.3-final -o srcrpt.1.lucene1.3.xml

A text file of paths to files may be used in place of a directory using the capital -I switch:

perl source_report.pl -I libwww579files -f keywords_2004_10_19.txt -o libwww5903_1.xml

Chunking                     Top

Since April, 2007, in an effort to speed up pre-acquisition legal Due Diligence, so-called "chunking", combined with "shadowing" has been a part of SrcRpt. Following is a brief description of the purpose of "chunked" scanning:

The problem: The acquisition target will not let the code to be scanned leave the premises, but the attorneys need to see and analyze the code scan.

Earlier solutions: 1. Have the attorney or other qualified representative travel to the premises to analyze the scan. 2. Have the target send the source code to us.

The problem with the earlier solutions: 1. Even if the lawyer or a qualified representative is able to travel go the premises, he/she doesn't have enough time/facilities to analyze the scan. 2. Difficult or impossible to get the target to send us the code before we have acquired it.

The solution: Create a "shadowed", "chunked" file system, in which all the directory names and file names are the same ("shadowed"), but the files are truncated ("chunked"). A variable amount of context is created around each hit in the scan (a variable amount negotiated with the target) and that amount only is written to the output file. Then the shadow file system is returned to the lawyer or other expert so that he/she can drill down from the scan into the source code as needed. The chunked files don't have enough technical information to compromise the intellectual property of the target, but they do have enough information to allow provenance/pedigree to be established.

The negotiation with the target over the amount of leading and trailing context around each hit is straightforward, once the target can see a few examples.

Chunking is accomplished using the -q switch and following the prompts at scan-time, or after the scan with the create_scan_dir.pl utility.

With either the -q option or the create_scan_dir.pl utility the user will see a series of prompts as follows:

Creating scan directory...Do you wish to create chunked files in the scan directory? [y] or [n] y

Creating chunked files in scan directory:

Enter the number of characters to be included before and after each hit in a chunk, or press <Enter> to accept the default of 500 characters.

250

Including 250 characters before and after each hit in a chunk. If this is correct enter [y]

y

Chunking 250 characters

...


The keyword/keyphrase file                  Top

The keyword file consists of one case insensitive keyword, key phrase or regular expression per line.  Comments begin with the character "#".

A keyphrase may be delimited by spaces or tabs.

Regular expressions are Perl 5 compatible and are indicated by "QQ" separated by a space at the beginning or end of a line..

 Keywords beginning or ending with "!" will be found if they occur at the beginning or end of a word, e. g., MS! will match "MSDN" and "MS-DOS".

Use of the “create_scan_dir.pl” script to assist in zipping for transmission.               Top

create_scan_dir has two purposes:

  1. It produces a folder containing only those source files which contained keywords, omitting binaries and files that did not contain keywords. So the new folder is smaller than the old one, and only contains files with keywords.

  2. It copies all the files necessary for someone else to analyze the scan, so that the resulting folder may be conveniently zipped.

The command line is:

perl create_scan_dir.pl -f <file name> -d <folder name> [ -c <comparison folder name> ]

where <file name> is the name of the SrcRpt XML file from the scan and <folder name> is the original folder scanned with the -d option to source_report.pl. For a diff scan use the -c option for the name of the comparison directory. The new folder will have a name: "SrcRpt.<nnnn>.<folder name> .

Example:

perl create_scan_dir.pl -f lucene.1.3.1.4.diff.xml -d lucene-1.3-final -c lucene-1.4-final

Where “lucene-1.3-final” and “lucene-1.4-final” are the folders containing the sources of a scan.

A new folder “SrcRpt.9042.lucene-1.3-final ” has been produced. This folder contains index.hta, the XML output of the scan, and all other files needed for analysis. The new folder may be zipped for transmission. Warning: The source directory or directories created do not contain all the source code, only code that contains keywords.
